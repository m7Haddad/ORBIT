# ORBIT MQTT Topic Convention

MQTT is used **only between the backend and devices**. Dashboard, Hermes, and n8n
never connect to the broker. Every topic below maps 1:1 to a `device_capabilities`
row and its REST endpoint ‚Äî same `{device_id}` (device UUID) and same `{capability}`
name in both.

| Concept | MQTT | REST |
|---|---|---|
| Capability state | `orbit/devices/{device_id}/{capability}/state` | `GET /api/v1/devices/{device_id}/capabilities/{capability}` |
| Capability command | `orbit/devices/{device_id}/{capability}/set` | `POST /api/v1/devices/{device_id}/capabilities/{capability}` |
| Availability | `orbit/devices/{device_id}/availability` | reflected in `is_online` / `last_seen` on `GET /devices/{device_id}` |

Topic segments are lowercase `snake_case`; `{device_id}` is the device UUID exactly
as stored in Postgres. The `orbit/` root prefix is configurable (Configuration
module), never hardcoded.

## 1. State ‚Äî `orbit/devices/{device_id}/{capability}/state`

- **Direction:** device ‚Üí backend
- **Retained:** yes (broker always holds last known value; backend also persists
  to `capability_readings`)
- **QoS:** 1
- **Payload (JSON):**

```json
{
  "value": 23.4,
  "ts": "2026-07-11T14:32:05Z"
}
```

- `value` type matches the capability's `data_type` (`float`/`int` ‚Üí number,
  `bool` ‚Üí boolean, `string`/`enum` ‚Üí string, `json` ‚Üí object).
- `ts` is device time (ISO 8601 UTC) when available; backend substitutes receive
  time if absent or obviously wrong.
- Composite example (`ac_control`):

```json
{ "value": { "mode": "cool", "setpoint_c": 22.0, "fan": "auto" }, "ts": "2026-07-11T14:32:05Z" }
```

## 2. Command ‚Äî `orbit/devices/{device_id}/{capability}/set`

- **Direction:** backend ‚Üí device (only the backend ever publishes here ‚Äî commands
  from users/Hermes/n8n all arrive via REST and are attributed in the audit log
  before publish)
- **Retained:** no (commands must never replay to a rebooting device)
- **QoS:** 1
- **Payload (JSON):**

```json
{
  "value": true,
  "request_id": "0d9c2c4e-8f1a-4b7e-9a53-2f6f0b1a7c11"
}
```

- `request_id` correlates the command with its `202` REST response and audit entry.
- The device confirms by publishing its new state to the `state` topic ‚Äî there is
  no separate ack topic. The backend pushes that state change to the dashboard
  over WebSocket.

## 3. Availability / LWT ‚Äî `orbit/devices/{device_id}/availability`

- **Direction:** device ‚Üí backend
- **Retained:** yes
- **QoS:** 1
- **Payload:** plain string `online` or `offline` (not JSON ‚Äî the LWT payload is
  fixed at connect time, so it cannot carry a dynamic timestamp; the backend
  stamps `last_seen` on receipt).
- Device publishes `online` (retained) immediately after connecting, and registers
  `offline` as its Last Will. The backend updates `devices.is_online` /
  `last_seen` and pushes the change over WebSocket.

## 4. Backend subscriptions

The backend (MQTT module) holds exactly two subscriptions:

- `orbit/devices/+/+/state`
- `orbit/devices/+/availability`

It publishes only to `orbit/devices/{device_id}/{capability}/set`.

## 5. Per-device credentials & ACLs

One broker login per device ‚Äî no shared credentials. Issued at manual device
registration (`POST /devices` returns them once) and revoked on device deletion.

- **Username scheme:** `dev-{device_id}` (e.g. `dev-4f1c9b2e-...`), matching
  `devices.mqtt_username`.
- **Password:** random 32+ char secret, generated by the backend, stored only as
  a hash on the broker side (Mosquitto password file or dynamic-security plugin).
- **Client ID:** must equal the username.
